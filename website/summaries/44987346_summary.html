The GitHub issue discusses the introduction of a new HTML element, <toggle-switch>, intended to provide a standardized, accessible, and customizable toggle switch control. The discussion covers various aspects, including use cases, accessibility considerations, styling, and naming. The main goal is to create a built-in HTML element that reduces the need for custom implementations and JavaScript-heavy solutions for toggle switches.

Use Cases: The proposed toggle switch element aims to cover scenarios where users need to switch between two states (on/off, yes/no, enable/disable). Examples include settings panels, preference controls, and form elements. The intention is to offer a semantic and accessible alternative to using checkboxes styled as toggle switches or relying on ARIA attributes and JavaScript.

Accessibility: A significant part of the discussion revolves around ensuring the new element is fully accessible. This includes proper ARIA attributes, keyboard navigation, and screen reader support. The element should provide clear visual and auditory cues about its state. The discussions involve how to handle labels, descriptions, and state indicators to meet accessibility standards.

Styling: The element's styling is another critical consideration. The goal is to allow developers to customize the appearance of the toggle switch using CSS while maintaining a consistent and recognizable visual structure. The discussion covers which parts of the element should be styleable and how to provide sensible default styles that align with different platforms and themes.

Naming: The name of the element itself is debated. Alternatives like <switch> are considered, but <toggle-switch> is preferred by some for its explicitness. The discussion also involves the naming of attributes and properties associated with the element.

Technical Details: The issue also delves into technical details, such as the element's attributes (e.g., checked, disabled), methods, and events. The interactions between the element and form submission are also addressed.

Alternatives: The discussion also acknowledges existing solutions, such as using styled checkboxes or custom elements. The proposal aims to offer a better alternative by providing a native, accessible, and customizable solution.

Key Points:

  • Proposal to introduce a native <toggle-switch> HTML element.
  • Aims to standardize toggle switch controls and improve accessibility.
  • Focus on providing a semantic alternative to JavaScript-heavy or styled checkbox solutions.
  • Extensive discussion on accessibility, including ARIA attributes and screen reader support.
  • Considerations for styling and customization using CSS.
  • Debate on the element's name and its attributes/properties.
  • Discussion of technical details, such as form integration and event handling.
