This article discusses the ongoing discussion within the kernel community regarding the proposed removal of the 'struct bio_vec' structure, often referred to as BIOVEC, from the kernel's public API. The BIOVEC structure is a fundamental part of the kernel's block I/O (input/output) subsystem, used to describe a contiguous region of memory involved in I/O operations. It's a key element in how the kernel handles data transfer to and from block devices like hard drives and SSDs.

The primary motivation behind the proposed removal is to improve the kernel's internal architecture and reduce the public API surface. Maintaining a stable public API is crucial for ensuring that out-of-tree kernel modules (drivers and other extensions) continue to function correctly across different kernel versions. However, a large and complex API can hinder development and refactoring efforts within the kernel itself. The BIOVEC structure, in its current form, is seen as an impediment to certain desired architectural changes.

Specifically, the article highlights that the current BIOVEC structure limits the ability to easily support larger block sizes and more flexible memory management schemes. The kernel developers want to move towards a more modern and efficient I/O handling system. Removing BIOVEC from the public API would allow for more freedom in modifying the underlying data structures and algorithms without breaking compatibility with external modules.

However, the removal of BIOVEC is not a simple task. Many out-of-tree drivers and other modules directly use the BIOVEC structure. Removing it would require significant changes to these modules, potentially breaking them. The article outlines the various approaches being considered to mitigate the impact of this change.

One approach involves providing alternative APIs that offer similar functionality to the current BIOVEC structure but are more abstract and less tied to the specific implementation details. This would allow out-of-tree modules to be adapted to the new system without requiring a complete rewrite. The new APIs would essentially act as a compatibility layer, providing a stable interface while allowing the kernel developers to modify the underlying implementation as needed.

Another aspect of the discussion revolves around the timing of the removal. Kernel developers need to provide sufficient time and resources for out-of-tree module maintainers to adapt their code. A phased approach, where BIOVEC is first deprecated and then eventually removed, is being considered. This would give developers ample warning and opportunity to migrate to the new APIs.

The article also touches on the technical challenges involved in providing suitable replacements for the functionality currently offered by BIOVEC. The new APIs need to be efficient, flexible, and easy to use. They also need to be carefully designed to avoid introducing new problems or limitations.

The discussion also involves concerns about performance implications. Any replacement for BIOVEC must not introduce performance regressions. The I/O subsystem is critical for overall system performance, so any changes must be carefully evaluated to ensure they do not negatively impact throughput or latency.

In summary, the proposed removal of BIOVEC from the kernel's public API is a complex and delicate operation with significant implications for both kernel developers and out-of-tree module maintainers. The goal is to improve the kernel's internal architecture and enable future enhancements, but this must be done in a way that minimizes disruption and maintains compatibility as much as possible.
